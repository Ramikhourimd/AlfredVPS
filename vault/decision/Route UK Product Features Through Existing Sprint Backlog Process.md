# Route UK Product Features Through Existing Sprint Backlog Process

## Context

UK clinic launch requires several product features: prescription module, scheduling system, report format adaptation, and questionnaire localization. The team needed to decide whether to create a dedicated UK development track or integrate UK requirements into the existing Israel-focused sprint process.

## Options Considered

1. **Dedicated UK development sprint** — Separate backlog and sprint for UK-specific features
2. **Integrated backlog** — Add UK features to the existing sprint backlog, prioritized alongside Israel work

## Decision

UK product features are routed through the existing sprint backlog process. Stav and Rami scheduled a 2 PM backlog meeting on Feb 15 specifically to add UK features (prescription, scheduling, report formats) to the shared backlog.

## Rationale

The team shares a single engineering team across Israel and UK. Creating a separate sprint would fragment developer attention. Instead, UK features compete for priority within the existing process, with the March 31 system launch providing urgency.

## Consequences

- UK features must compete with Israel feature requests for engineering bandwidth
- No guaranteed allocation of engineering time for UK-specific work
- Risk of UK features being deprioritized when Israel operational issues arise
- Stav flagged that prescription and scheduling are "more complex than other backlog items" — they may crowd out other priorities
