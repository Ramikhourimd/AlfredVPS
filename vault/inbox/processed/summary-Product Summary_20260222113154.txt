# סיכום פגישה: לוגיקת דוחות, אינטגרציית API וארכיטקטורת AI לסיכומים
## נקודות מרכזיות
- סטנדרט לוגיקת אירועים לדוחות: אישור סיכום מזוהה דרך approve (מכבי) או send (מרכזים אחרים); save אינו אישור. פעילות אמיתית החלה סביב 10 במרץ, לפני כן רובם בדיקות.
- מדידת זמני עריכה: מהימן רק בריל-טיים; אירועי “modified” ב-Google Docs אינם נקודת התחלה תקפה.
- איכות נתונים: קיימים רעשים וזיהומים (זמני המתנה לא סבירים, גילאים בלתי אפשריים); נדרש סינון וטיוב.
- ארכיטקטורת נתונים וביצועים: המלצה לשכבת Read-Only/Shadow DB לשאילתות לא-ריל-טיים ולהפחתת עומס על הפרודקשן; שימוש בפייג’ינג לקריאות נפחיות.
- API-first ושקיפות: העדפה חזקה לממשקי API קבועים במקום אקספורט ידני; חלוקה לקריאות Low-Level לפי ישות (Patient, Meeting, Questionnaire).
- טריגר להרצות: Webhook כמנגנון יעד, עם פתרון זמני דרך Start API עד לייצוב ה-Webhook; העשרת פרמטרי התחלה (user ID, clinic, meeting type/ID, meeting ID).
- סטנדרטיזציית פלט: מעבר לפורמט JSON מובנה עם היררכיית סקשנים אחידה; אפשרות לפלט HTML מובנה לפי צורך.
- זרימות שירותי AI: להתחיל אורקסטרטור פשוט ללא MCP ולהשאיר אופציה להוספת MCP בהמשך (בעיקר לצ’אט/אורקסטרציה מתקדמת).
- דוחות קליניקה: צורך ב-API חודשי לכל הפגישות (כולל Cancel/No-show), שדות עשירים ויכולת פילטור; קישור למחירים דרך DWH/BigQuery.
- תפעול תמלול בפסיכיאטריה: הפעלת Teams Captions בזמן אמת והפעלת Transcript לסוף פגישה כגיבוי; בחינת רקורדר מקומי ברקע להפחתת חיכוך.
## בעיות/חסמים שעלו
- חוסרים בדוח שמוליק והבדלי לוגיקה בין קליניקות (#data_gaps).
- אי-יכולת למדוד זמן עריכה מחוץ לריל-טיים (#measurement_limitations).
- נתוני קצה/רעשים (זמני המתנה קיצוניים, גילאים בלתי אפשריים) (#data_quality).
- עומסי מערכת בעת משיכות נתונים גדולות (#system_load); צורך בתזמון מחוץ לשעות עומס.
- אתגרי אבטחה ופרטיות בריבוי מקומות אחסון; העדפה למינימום שמירה של נתונים רגישים.
- חוסר יציבות ב-Webhook; שימוש זמני ב-Start API.
- גישה חלקית לסטים ב-DWH והיעדר סטנדרטיזציה מלאה; צורך בפייג’ינג למניעת כשלים.
- מפגשים פתוחים/טסטים שלא נסגרו; קושי לקבוע סטטוס ללא פעולה ידנית.
## רעיונות/פתרונות שנדונו
- אימוץ לוגיקת אישור אחידה: approve/send כאישור, התעלמות מ-save.
- סינון רעשים: הסרה מחוץ לספים סטטיסטיים/סטיות תקן; טיוב גיל והסרתו עבור ערכים לא סבירים.
- שכבת Read-Only/Shadow DB: הורדת עומסים, תמיכה בשאילתות אנליטיות, ותזמון משיכות מחוץ לשעות שיא.
- עיצוב API Low-Level: Get Patient, Get Meeting (Transcript/Summary/Diagnosis/Prescription), Get Questionnaire; תמיכה בפייג’ינג ותקרות פיילוד.
- טריגרים והרשאות: שימוש ב-Webhook/Start API עם מנגנון Auth ולוגים מפורטים; העשרת פרמטרי ה-Start.
- סטנדרט פלט JSON: מבנה סקשנים אחיד; תמיכה באלטרנטיבת HTML מובנה.
- APIs ייעודיים: 
  - החזרת סיכום מקורי וסיכום ארוך לפי Meeting ID להשוואות/Scoring.
  - דוח חודשי לכל הפגישות (כולל סטטוסים, סיבות ביטול, זמני יצירה/קביעה, מטפלים).
- תמלול פסיכיאטריה: הפעלת Captions עם שפה מוגדרת; הפעלת Transcript כגיבוי; בחינת רקורדר מקומי ברקע.
- MCP Orchestrator: לשקול בהמשך לצמצום עצי החלטה ידניים ולהרחבת שימושי צ’אט.
## משוב/הצעות מהמשתמשים
- שקיפות ועדכניות: העדפה ל-API קבועים במקום אקספורט ידני; לוגים מפורטים ובקרות גישה.
- אחידות בין קליניקות: לזהות אישור גם ב-send, לא רק approve.
- ביצועים תחת עומס: תמיכה בשאילתות יומנים/פגישות בהיקפים גדלים.
- דוחות חודשיים עשירים: לכלול Cancel/No-show ושדות אנליטיים (סיבות ביטול, יוצר/מועד יצירה, מטפל).
- הפחתת חיכוך תפעולי: תמלול אוטומטי ככל הניתן; הימנעות מריבוי מקומות אחסון לנתונים רגישים.
- קונפיגורביליות ללקוחות חו"ל: שאלונים/פרומפטים ניתנים לקונפיגורציה ללא שינויי קוד חריגים.

## לוחות זמנים/שלבים
- תחילת נתונים אמיתיים: סביב 10 במרץ.
- משיכות נתונים חודשיות: יעד לקבלת נתונים בראשון או שני לחודש דרך API.
- אין תאריכים נוספים מפורשים; חלונות להרצות מחקר/משיכות גדולות יוגדרו לאחר בדיקת עומסים.
## הצעות למעקב לפגישות הבאות
- סקירת סט קריאות API מאושר: דוגמאות JSON, פרמטרים, פייג’ינג, וזרימת End-to-End (טריגר → איסוף → סיכום → החזרה/Push).
- דוח עדכני עם סינון רעשים: השוואת לפני/אחרי והשלכות על מדדים.
- עדכון בדיקת עומסים והגדרת חלונות זמן למשיכות גדולות; סטטוס Read-Only/Shadow DB.
- הדגמת פיילוט: השוואת סיכום מקורי מול סיכום ארוך (איכות/פערים).
- טיפול בחוסרים בדוח שמוליק, כולל גילאים לא סבירים וניקוי מפגשים פתוחים.
- החלטה סופית על סטנדרט פלט JSON וסקימה; מיפוי זרימות דטרמיניסטיות מול אג’נטיות (סיכום פסיכיאטר, אג’נדה, צ’אט).
- בחינת מודל MCP בהמשך לשילוב Chatbot; אימוץ מדיניות אבטחה/לוגים ולכידות ברחבי ה-APIs.
