## הערות פגישה
### ארכיטקטורת מערכת, API וניהול נתונים
#### בניית Orchestrator וארכיטקטורת API
הדיון התמקד בתכנון ארכיטקטורת מערכת AI גמישה. רמי הציע לבנות Orchestrator מסוג MCP (Multi-Agent/Connector Platform) שינהל באופן דינמי קריאות ל-API בהתבסס על סוג הפגישה והמטפל (`meeting ID`, `meeting type`, `user ID`, `clinic`). זאת במטרה להימנע מעץ החלטות קשיח ולאפשר למערכת לבחור את הסוכן (Agent) וה-API הנדרשים באופן אוטונומי. עלתה שאלה האם ליצור שירות מרכזי אחד או נקודות כניסה נפרדות לפונקציות שונות (אג'נדה, סיכומים וכו'). סוכם כי במקום קריאת API אחת שתחזיר את כל המידע, עדיף לפצל למספר קריאות נפרדות (למשל, פרטי מטופל, פרטי פגישה, תמלול), וה-Agent יקבל מידע גולמי ויעבד אותו. Speaker 3 הסכים לספק את כל קריאות ה-API הנדרשות כדי לאפשר לרמי להריץ מחקרים וסיכומים באופן עצמאי.
#### סטנדרטיזציה של פלט ופורמט נתונים
הוסכם על הצורך בסטנדרטיזציה של מבנה הפלט מה-AI כדי להקל על העיבוד. המטרה היא לקבל תמיד פלט מפורק לסקשנים במבנה JSON היררכי קבוע, גם אם יש רק סקשן אחד, בדומה למה שנעשה בפרויקט הפסיכיאטריה. בנוסף, רמי ביקש API שיספק גם את הסיכום המקורי וגם סיכום ארוך לצורך ביצוע fine-tuning והשוואה בין גרסאות.
#### אבטחת מידע ואחסון
עלתה דאגה לגבי אבטחת מידע באחסון סיכומים ותמלולים רגישים. הוסכם כי שימוש ב-API בטוח יותר מאשר אחסון ישיר בבסיסי נתונים כמו BigQuery או Google Drive, שכן כל מיקום אחסון נוסף דורש אמצעי אבטחה ובקרת גישה. הוזכרה האפשרות לצמצם בעתיד את השימוש ב-Drive כאמצעי אחסון.
### ביצועי מערכת, עומסים וניהול נתונים
#### טיפול בעומסים על המערכת
הועלה חשש מעומסים על המערכת, במיוחד בעת הרצת קריאות API על אלפי מטופלים במקביל. כפתרון, הוצע לשכפל את בסיס הנתונים הראשי (Xano) לגרסת "read-only" שתשמש לשאילתות מחקר ובדיקה, ובכך למנוע פגיעה במערכת האופרטיבית. כמו כן, סוכם כי פעולות גדולות יבוצעו מחוץ לשעות העומס. כדי למנוע עומסים בשליפת נתונים גדולים, הוצע לעבור לשיטת "עמודים" (paging), למשל 50 רשומות בכל פעם, או לבצע קריאות API נפרדות לכל מטפל.
#### ניתוח דוחות וטיפול בחריגות
במהלך בחינת דוח סיכומים של שמוליק, זוהו ישויות שונות לאישור סיכום פגישה במערכות שונות (למשל, `Summary Approved` במכבי לעומת `Send` בקליניקות אחרות). סוכם כי `send` שקול ל-`approve`, והלוגיקה נכונה. כמו כן, זוהו נתונים חריגים בדוח, כגון זמני המתנה של 300 ימים וגילאים לא הגיוניים (מעל 93). הוחלט להתעלם מזמני ההמתנה החריגים ולהסיר את נתוני הגיל עבור 22 המטופלים עם גילאים שגויים. בנוסף, זוהתה בעיה של פגישות שנשארות במצב "פעיל" גם לאחר מועדן, בעיקר בחשבונות טסט, אך לא נקבע פתרון לטיפול במקרים האמיתיים.
#### גישה לנתונים ו-Data Warehouse (DWH)
רמי ביקש גישה לטבלאות נוספות ב-DWH, כולל נתוני תשלומים לפסיכיאטרים. הוסכם שרמי יקים Dataset משלו ויעלה לשם את טבלת התשלומים. כמו כן, רמי ביקש API שיספק דוח חודשי מפורט של כל הפגישות, כולל סוג, משך, סטטוס (כולל ביטולים ו-no show), וקבוצת גיל. Speaker 3 הסכים לספק API כזה עם כל המידע הקיים.
### שיפורים תפעוליים ופיתוח עתידי
#### מדידת זמן עריכת סיכום
רמי הביע עניין במדידת הזמן שלוקח לפסיכיאטר לערוך סיכום פגישה. המסקנה הייתה שלא ניתן למדוד זאת רטרואקטיבית עבור פגישות שאינן ב-Real Time, מכיוון שהמערכת הישנה לא תיעדה את פתיחת מסך העריכה.
#### שיפור תהליך התמלול
נדון הקושי בכך שפסיכיאטרים צריכים להפעיל ידנית כתוביות (Captions) ב-Teams כדי לאפשר תמלול בזמן אמת. הוצעה אפשרות להשתמש בכלי צד שלישי שמקליט את השמע מהמחשב ברקע באופן אוטומטי. צוין כי התמלול המלא (Transcript) שנוצר בסוף הפגישה איכותי יותר מהכתוביות בזמן אמת.
#### התאמת המוצר לשווקים חדשים
נדון האתגר בהתאמת המוצר, המותאם כיום לישראל, לשווקים בינלאומיים (כמו אנגליה). Speaker 3 ציין כי הדבר דורש שינויים משמעותיים והשקעה גדולה, והופך מודל תמחור נמוך ללא כלכלי. בניית מוצר גנרי דורשת מאמץ פיתוח גדול משמעותית, והאתגר הוא לאפשר ללקוח להגדיר את המערכת בעצמו.
